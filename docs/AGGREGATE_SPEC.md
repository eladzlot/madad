# AGGREGATE_SPEC

Design document for the **Aggregate** surface (`/aggregate/`, סיכום) — a
clinician-facing tool for aggregating Madad PDFs into longitudinal
trajectories.

This document captures *what the system is and why it is that way*. It is
maintained as the system evolves. For build phasing and task ordering see
the implementation plan.

Sibling docs: `BEHAVIORAL_SPEC.md`, `COMPOSER_SPEC.md`,
`IMPLEMENTATION_SPEC.md`.

---

## 1. Purpose and scope

Madad's patient flow ends with a single PDF: the canonical record of one
session. Clinicians collect these PDFs over time. Aggregate is the surface
that turns a collection of PDFs into a chart of change.

**Aggregate is read-only and stateless.** It does not store data, does not
have accounts, and does not transmit PDFs anywhere. The clinician drops
PDFs into the page; the page reads structured data embedded in those PDFs;
the page renders charts. Closing the tab discards everything.

**Aggregate is a clinician-side tool, not part of the patient flow.** The
patient never sees Aggregate. The PDF the patient downloads is the same
PDF Aggregate reads — Aggregate is parasitic on an artefact that already
exists.

### 1.1 What Aggregate is not

- Not an EHR. No long-term storage.
- Not a clinical decision support system. The chart is descriptive, not
  prescriptive.
- Not a sharing platform. PDFs reach Aggregate through whatever channel
  the clinician already uses (email, file system, USB stick); Aggregate
  doesn't transport them.

---

## 2. Position in the system

The product has four surfaces:

| Surface | Path | Audience | Function |
|---|---|---|---|
| Patient app | `/` | Patient | Complete a battery, download a PDF |
| Composer | `/composer/` | Clinician | Build a session URL |
| Aggregate | `/aggregate/` | Clinician | Aggregate PDFs into charts |
| Landing | `/landing/` | Both | Marketing, entry point |

The three clinician surfaces (Composer, Aggregate, Landing) share a
top-level navigation component. The patient app does not — patients in
the middle of a clinical battery should not see clinician-facing nav.

The PDF is the *integration boundary* between the patient flow and
Aggregate. Everything Aggregate shows is reconstructed from data embedded
in the PDFs the clinician feeds it.

## 3. Data model

### 3.1 The PDF as input

Every PDF generated by Madad carries an embedded JSON file named
`data.json`, attached using pdfmake's `files` dictionary. The file
appears in PDF viewers' attachment panels and is extractable by any
PDF-aware parser.

The embedded payload is the canonical machine-readable representation of
a session. The visible PDF is the canonical human-readable
representation. Both come from the same source data (`sessionState`,
`config`, `session`). They do not diverge.

### 3.2 Envelope schema

```jsonc
{
  "schemaVersion": 1,
  "generatedAt":   "2026-04-12T08:33:00Z",
  "appVersion":    "0.9.3",
  "pid":           "TRC-2025-000123",
  "name":          "...",
  "instruments":   [
    {
      "questionnaireId": "phq9",
      "title":           "שאלון דיכאון (PHQ-9)",
      "configFile":      "standard"
    }
  ],
  "sessionState": {
    "answers": { "...": "..." },
    "scores":  { "...": "..." },
    "alerts":  []
  }
}
```

`schemaVersion` is the **envelope version**. It bumps when the top-level
shape changes. Aggregate must understand every shipped envelope version
forever — old PDFs continue to work in future Aggregate builds. v1 → v2
upgrades happen via explicit migration code, not deprecation.

`generatedAt` is the timepoint that puts a session on the X-axis.
`appVersion` is forensic only.

`pid` and `name` are session metadata, copied from `session`. Both may be
`null`; Aggregate treats both as opaque strings.

`instruments` is metadata about which questionnaires the session
contained — title, ID, source config file. Aggregate uses this for
routing and labelling. There is **no per-instrument version**. The
project treats questionnaire changes as out-of-band events handled
manually if and when they occur.

`sessionState` is the unmodified state object the patient app already
produces. Item-level answers are included because Aggregate may
eventually graph them (out of v1 scope, but the data is captured).

### 3.3 What is *not* in the payload

- Cryptographic signature. The clinician trusts the patient who shared
  the PDF; signing would be theatre.
- Encryption. Encryption requires a key, a key requires accounts,
  accounts kill the privacy model. The PDF is plaintext-equivalent for
  anyone who has it.
- Free-form clinician notes. Aggregate is read-only; if the clinician
  wants notes they live in their own systems.

### 3.4 Versioning policy

Two version numbers exist in the system:

- **Envelope version** (`schemaVersion`). Owned by the PDF generator.
  Bumped when the JSON shape changes. Forward-compat is a project
  commitment.
- **Per-questionnaire content** is **not versioned**. Questionnaire
  scoring formulae, item sets, and subscale definitions are treated as
  stable. If a bug fix or correction necessitates a change, it is
  handled as a one-off event and communicated out-of-band — Aggregate
  does not detect or guard against it.

The asymmetry is intentional. Envelope shape changes whenever the
project decides to evolve the data model; questionnaire content is
fixed by external research and changes rarely if ever.

## 4. Patient identity

Aggregate does not group, dedupe, or cluster PDFs. The clinician is
responsible for upload hygiene.

By default Aggregate aggregates *every uploaded PDF into a single
trajectory*, regardless of `pid`. The clinician supplies the right files;
Aggregate charts what it's given.

`pid` is exposed as a **filter**, not a grouping primitive. The filter
bar lists every distinct `pid` seen across uploaded files (plus "all"
and "no pid"). Selecting one narrows the view.

This design accepts a real failure mode: a misfiled PDF will be silently
included in the chart. A PHQ-9 jumping from 6 to 22 might be a different
patient, not a relapse — Aggregate will not help distinguish. This
matches the rest of Madad's posture: trust the clinician, no validation
theatre.

## 5. Chart semantics

### 5.1 What a chart shows

For each instrument with a quantitative `total` score, Aggregate renders
one chart:

- X-axis: real dates from each PDF's `generatedAt`. Time flows
  right-to-left (older on the right) to match Hebrew RTL reading order.
- Y-axis: total score, scaled 0 to instrument max.
- Severity bands as horizontal background fills (when applicable).
- A single line connecting session points, with markers at each session.
- A baseline marker on the first session (visually distinct).
- An optional dashed RCI threshold line (when psychometrics available).

Subscales are available as additional togglable lines via a per-chart
legend. Total is on by default; subscales are off by default.

### 5.2 Severity bands and screening cutoffs

Each instrument's `interpretations.ranges[]` carries a `type` field:
either `"severity"` or `"screening"`.

- ≥3 ranges of `"severity"` → render as horizontal background bands,
  one band per range, coloured by severity (warm tones, lifted from the
  PDF's existing palette).
- A single `"screening"` cutoff → render as a single solid horizontal
  line at the cutoff value.
- No `interpretations` block → no overlay.

Severity bands fade out when the total series is hidden (subscales-only
view). Bands describe total-score ranges; applying them to a subscale
would mislead.

### 5.3 Reliable Change Index

Each instrument's config may carry an optional `psychometrics` block:

```jsonc
"psychometrics": {
  "reliability": 0.89,
  "sd": 5.7,
  "source": "Kroenke et al. 2001"
}
```

When present, Aggregate renders a dashed horizontal line at
`baseline.total ± rciThreshold` (one above, one below):

```
RCI = 1.96 × sd × √(2 × (1 − reliability))      [95% confidence]
```

The line is the **goal**. The patient's trajectory crossing it is the
visible event. RCI is computed against baseline (the first session in
the uploaded set), not point-to-point — the clinical question is
"meaningfully different from where they started," not "different from
last session."

The RCI line is visually distinguished from severity cutoffs by both
line style (dashed vs solid) and colour (cool teal — the
landing-page brand colour — vs warm severity tones). A right-anchored
inline label "סף שינוי מהימן" identifies it.

When psychometrics are absent, the line simply doesn't render. There is
no "RCI unavailable" message — the chart silently degrades. The
`psychometrics` block is shipped only on instruments where the
literature supports it.

Baseline = first uploaded session. There is no override UI in v1.

### 5.4 The visible window

The default visible window is the **most recent 5 sessions**. With more
than 5 uploaded:

- The chart paginates horizontally with arrow controls.
- A small badge displays "מציג 6–10 מתוך 12" or similar.
- The default view is anchored to the most recent end.

With ≤5 sessions, the chart spans the full available width; X-axis
spacing reflects real time gaps between sessions. Empty right side
(when only 1–2 sessions exist) reads visually as "future."

Brush/zoom across the full history is a v2 feature.

### 5.5 Non-quantitative items

Four categories of items have no `total` and are out of scope for v1
graphing:

- Idiographic rating scales (TOP-3 style)
- Binary screeners (DIAMOND-SR style)
- Free-text responses
- Worksheets (anger log, future CPT worksheets)

For these, Aggregate shows a "raw data, not graphed" list grouped by
instrument, with a per-session view link. The data is captured in the
embedded JSON regardless, so v2 graphing is a UI question, not a schema
migration.

### 5.6 Interaction

- Hover or focus on a point → tooltip with date, total, all subscales,
  alerts.
- Click or Enter on a point → a slide-in detail panel with the full
  session breakdown, all instruments completed that day, and a download
  link for the underlying PDF (which Aggregate kept in memory as a Blob).
- Arrow keys move focus across points within a chart.
- A "view as table" button below each chart toggles a tabular rendering
  of the same data — primary screen-reader experience and useful as a
  numeric reference.

### 5.7 Bad-PDF handling

Three failure modes, all per-file, none aborts the page:

| Mode | Behaviour |
|---|---|
| Not a Madad PDF (no attachment) | File-level warning in the upload list; other files continue to process |
| Unknown envelope version (newer than this build) | File-level warning; file ignored |
| Malformed payload (parse error, shape error) | File-level warning; file ignored |

Nothing ever silently aborts. Nothing crashes the page. The clinician
sees per-file status in the upload list and adjusts.

## 6. Image export

Charts can be exported as PNG (primary) or SVG (secondary) for
out-of-band sharing — typically with a colleague, supervisor, or
referring physician for an outcome judgement.

**Always on the image:** instrument title, the chart itself, date range,
generation timestamp, "מדד" footer.

**Never on the image:** patient name, free-text patient responses.

**Optional, default off:** `pid` (some recipients expect it as a
cross-reference key, others treat any identifier as PHI), subscale
lines.

PNG sizing: ~1600 × 1000 px, rendered at 2× density.

The export is rendered entirely in the browser via SVG → canvas →
`toBlob`. No data leaves the device.

## 7. Privacy posture

Aggregate matches the rest of Madad:

- No analytics on `/aggregate/`.
- No external network requests when reading PDFs.
- No `localStorage` or `sessionStorage` of patient data.
- Strict same-origin (CSP inherited from `index.html`).
- Closing the tab discards everything.

The page tells the clinician this explicitly, framed as a feature:

> הקבצים נטענים בדפדפן שלך בלבד. סגירת הכרטיסייה מוחקת אותם.

Persistent storage (encrypted IndexedDB, passphrase-protected caching) is
not implemented in v1. It may be revisited if real clinician feedback
demands it.

## 8. Composer changes

The Composer is the only other surface that changes for Aggregate:

- A soft prompt: *"Planning to track this patient over time? Add a
  clinic ID."* `pid` does not become required — single-shot use must
  stay frictionless.

A future refinement (not v1) is a "tracking battery" preset for
common routine outcome monitoring instruments (PHQ-9 + GAD-7 + WSAS).

---

## 9. Out of scope for v1

Decided as not-now. Revisiting requires updating this document.

- Patient-side aggregation (patient sees their own aggregate).
- Self-contained "latest PDF embeds prior history" pattern.
- Encrypted IndexedDB caching.
- Animated chart transitions.
- Brush/zoom on the X-axis.
- Idiographic / binary / free-text / worksheet trajectory **graphs**
  (their data is captured now).
- Baseline override UI.
- Multi-instrument overlay on shared axes ("% of range" view).
- Annotations on the chart.
- Patient name on shared images, ever.
- Signing or encryption of embedded JSON.
- Per-instrument versioning and conflict detection.

Aggregate requires changes to existing specs and configs:

- `CONFIG_SCHEMA_SPEC.md` — interpretation ranges gain `type:
  "severity" | "screening"`; questionnaires gain optional
  `psychometrics`.

  documented; `ENVELOPE_VERSION` constant introduced.
- `BEHAVIORAL_SPEC.md §3, §8, §10` — Aggregate exists as a clinician
  surface; PDFs are now also Aggregate input; privacy wording updated.
